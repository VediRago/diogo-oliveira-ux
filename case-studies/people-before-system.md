# People Before System

*One of the applied methods behind [the repository's principles](../PRINCIPLES.md).*

## Problem

A system gets built in one order. A reader encounters it in a different order. Explaining a system the way it was built forces the reader to learn architecture before they have a reason to care about it.

## Method

Present information in the order a reader actually discovers it, not the order it was constructed in. Build character before system. Let the reader form attachment first.

An onboarding flow built the same way presents a feature in the order a user runs into the need for it, not the order engineering built the underlying system.

---

## Case: Narrative (Noeme)

Early development began with images and outcomes, not structure. A hospital girl. Pale grey eyes. A collapse. The architecture explaining why those moments belonged together came after.

The recommended reading path follows the same order: character, then consequence, then system.

→ See: [People Before System](../vault/people-before-system/people-before-system.md)

## Case: Worldbuilding (Erit)

A character does not need atmosphere. Atmosphere is not added until behavior has repeated enough to produce it. History is not added until atmosphere has lasted long enough to become durable. Quest logic is not added until history has created a playable problem.

Each layer waits for the one before it.

→ See: [Worldbuilding Method Hierarchy](../vault/people-before-system/worldbuilding-method-hierarchy.md)
