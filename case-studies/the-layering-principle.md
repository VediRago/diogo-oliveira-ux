# Layering as a Method

*One of the applied methods behind [Consistency](./consistency.md) — the goal this exists to serve.*

## The Method

Layering is a method for organizing and deploying information, used repeatedly across this body of work to achieve one goal: consistency. Whoever is on the other end — a reader, a player, a portfolio visitor — should always know where they are and what to expect next, regardless of how much depth exists underneath what they're currently looking at.

The method isn't one fixed rule applied identically everywhere. It flexes depending on what the problem needs. Sometimes layers have to be built in a strict sequence, because a later layer literally cannot exist without an earlier one. Other times, all the layers already coexist, and the only question is how much of that depth a given reader chooses to go looking for.

What stays constant across every case is the goal, not the mechanic: whatever layer someone is standing on should feel complete and consistent on its own terms.

---

## Case: Content Depth (Worldbuilding)

Sequence-dependent by design: nothing gets added until what already exists has organically demanded it. A character doesn't get atmosphere until behavior has repeated enough to earn it. A place doesn't get history until atmosphere has persisted long enough to become durable. The layers can only be built in order.

## Case: Scene Blueprint (Production)

The same sequence-dependent logic at the production level: a global rule set has to exist before a scene can reference it, and the scene has to exist before a screenplay can execute it. Each stage is impossible to write correctly without the one before it already in place.

## Case: Story Structure (Fragments)

Where the two mechanics meet. In development, character came first; deeper structure was built afterward, specifically to explain why the character moments belonged together. In the finished story, all layers exist at once — a reader who only experiences the surface gets a complete story, and the deepest layer is never explained directly, only felt as coherence.

→ See: [Layered Narrative System](https://github.com/VediRago/diogo-oliveira-portfolio/blob/main/case-studies/fragments-layered-narrative.md)

## Case: Simulation (Twine Prototype)

Designed as one interlocking system, not built in sequence — but at runtime it presents itself with the same optional depth. The surface shows only prose and consequence. The mechanical layer underneath is a separate, optional view, one click away, never forced on anyone.

→ See: [Designing the Invisible System](./invisible-system.md)

## Case: Portfolio Architecture

The portfolio repositories apply the same optional-depth pattern to their own structure — a README stands completely on its own, and a Vault holds what's underneath it. This is one instance of the broader consistency principle this method serves; see [Consistency](./consistency.md) for the full picture.

---

## Why This Matters

Five different problems — a story, a documentation system, a piece of production tooling, an interactive simulation, and a portfolio's own architecture — get solved with the same underlying method, because the goal in each case is identical: whoever depends on a given layer should never have to encounter it breaking its own rules. The mechanic changes with the problem. The consistency doesn't.
