# Layer 2 Blueprint

## Purpose

A Layer 2 repository explains how a professional domain is approached.

It does not present finished work directly. It explains the methods, principles, and systems behind the work, then links to evidence that proves those methods were used.

Layer 2 exists so a reader can understand the thinking behind a discipline before deciding whether to inspect individual projects or products.

## Core Question

Every Layer 2 repository answers one question:

> How is this work approached?

If a repository is mainly answering who the author is, it belongs in Layer 1.

If it is mainly presenting finished work, it belongs in Layer 3.

If it cannot be generalized into a professional method without distortion, it may belong in Layer 4.

## Required Structure

A Layer 2 repository contains:

1. **README** — orients the reader, states the domain, and provides the reading path.
2. **Principles** — defines the rules governing the repository's documents, navigation, and evidence.
3. **Case Studies** — each case study explains one transferable method.
4. **Supporting Files** — provide focused depth without interrupting the case study.
5. **Vault Evidence** — stores artifacts that prove the case study happened.

The default reading path is:

```text
README
  ↓
Principles
  ↓
Case Study
  ↓
Supporting Files
  ↓
Vault Evidence
```

## Case Study Rule

A Layer 2 case study explains a method, not a finished product.

The case may come from a project, story, prototype, production system, or portfolio decision, but the case study's job is to extract the transferable thinking.

A case study belongs in Layer 2 when the reader can take away an approach that could apply beyond the original artifact.

## Evidence Rule

Layer 2 does not rely on summary alone.

Every method should be supported by real artifacts: blueprints, drafts, inventories, templates, maps, process notes, production files, or other evidence that shows the method being used.

Evidence lives in the Vault belonging to the case study that frames it.

## Navigation Rule

A Layer 2 repository must remain readable at multiple depths.

A reader should be able to stop after the README, after Principles, after one case study, or after following evidence into the Vault and still understand something complete.

Every supporting file and Vault file returns to the document that owns it.

## Consistency Rule

Layer 2 repositories share structure because their job is similar: explain method through principles, cases, and evidence.

The content may differ by discipline, but the reader should not have to relearn how the repository works each time.

## Completion Checklist

A Layer 2 repository is complete when:

- the README clearly states the domain and reading path
- Principles define the repository's rules
- every case study explains one method
- every case study links to real supporting material or evidence
- supporting files answer one focused question each
- Vault files return to their owning case study
- no document tries to answer more than one layer question
- the repository can be understood independently while linking naturally to the wider portfolio
